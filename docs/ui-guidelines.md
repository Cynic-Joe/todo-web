# UI 设计规范

这份文档是当前正式版界面的设计基准。后续 agent 只要涉及页面、组件、排版、视觉或文案改动，都应先遵循这里的规则，再落代码。

## 1. 整体方向

- 视觉气质：现代、简洁、克制，接近 Apple 式的清爽工具感。
- 可保留轻微纸页、便签、记事本质感，但只能作为背景气氛，不能主导画面。
- 不要国风、古风、厚重复古纸张、旧报刊或“展示型概念页”审美。
- 页面应该像一个干净的记录本，而不是一个带讲解的设计提案。

## 2. 页面结构

- 采用单列主工作区。
- 顶部只保留轻量入口和必要操作，不做大 Hero 区或大摘要区。
- 不再恢复以下类型的独立板块：
  - “纸黄待办”式品牌主视觉
  - “工作台摘要”式总览区
  - “收支侧栏”式辅助摘要区
- 主界面优先展示当前正在操作的功能区内容，而不是解释页面结构。

## 3. 导航与信息架构

- 四个主功能固定为：`待办`、`搁置`、`完成`、`记账`。
- 导航用中文章节切换，不用英文副标题，也不用厚重看板式标签按钮。
- 界面文案一律中文优先。
- 只有在必须指向第三方服务时，才允许保留 `GitHub`、`Gist` 这类专有名词。

## 4. 文案规则

- 禁止“设计说明型文案”。
- 不要写类似下面这种内容：
  - “沿用同一套结构”
  - “不抢占主操作区”
  - “统一按钮体系”
  - “延续同一套卡片语言”
- 文案只回答用户真正需要知道的事：
  - 这里是什么
  - 现在有没有内容
  - 下一步能做什么
- 所有中文必须正常 UTF-8，禁止乱码、残缺标点或错误币种符号。

## 5. 组件语言

- 卡片、输入框、按钮、空状态采用同一套轻量视觉语言。
- 任务、搁置、完成、记账共用统一容器体系，不为每个区块单独发明一套视觉。
- 通过状态、金额、时间、轻微色彩区分不同内容，而不是通过复杂装饰区分。
- 图标只作为辅助识别，不能压过文字层级。

## 6. 记账区规则

- 记账区是“流水页”，不是“摘要看板”。
- 汇总信息只能做紧凑统计，不能重新长成大面积摘要卡墙。
- 录入区和记录列表是主角。
- 收入和支出在结构上保持统一，只通过金额符号、颜色和标题做区分。

## 7. 交互与状态

- 操作应保持直接：新增、删除、完成、搁置、恢复、记收入、记支出都应在当前语境内完成。
- 空状态要简短，不讲设计理念，只提示下一步动作。
- 设置抽屉可以保留，但应作为次级入口，不应抢占主页面视觉重心。

## 8. 禁止项

- 不要恢复英文副标题，如 `Active`、`Archive`、`Review`、`Ledger`。
- 不要恢复展示型说明卡、概念说明卡、侧栏摘要卡。
- 不要引入风格过重的国风、古风或复古排版。
- 不要为了“设计感”增加不承担任务价值的文案和层级。
